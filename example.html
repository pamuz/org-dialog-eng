2. Python Data Processing Example

I'm working on a Python script to process data from a CSV file. 

import csv

def process_data(filename):
    with open(filename, 'r') as f:
        reader = csv.reader(f)
        data = list(reader)
    return data

Can you help me add error handling to this function? I want to handle:

  1. File not found errors
  2. CSV parsing errors
  3. Empty files

Here's an improved version with comprehensive error handling:

```python import csv import os

def processdata(filename):

if not os.path.exists(filename): raise FileNotFoundError(f"File '{filename}' not found")

try: with open(filename, 'r') as f: reader = csv.reader(f) data = list(reader)

if not data: raise ValueError(f"File '{filename}' is empty")

return data except csv.Error as e: raise ValueError(f"Error parsing CSV file: {e}") except Exception as e: raise RuntimeError(f"Unexpected error processing file: {e}") ```

Key improvements:

  1. File existence check: Uses `os.path.exists()` before opening
  2. CSV parsing errors: Catches `csv.Error` exceptions
  3. Empty file check: Validates that data was actually read
  4. General error handling: Catches unexpected errors with proper context

6. Notes About This Workflow

The beauty of this approach is that:

  1. Context is preserved: Each prompt has access to all previous conversation
  2. Documentation included: Your notes and code are right alongside the AI responses
  3. Branching conversations: You can re-execute earlier prompts to explore different paths
  4. Version control friendly: Everything is in plain text, works great with git

6.1. How to Use

  1. Enable org-ai-prompt-mode in your org buffer
  2. Write your notes, code, and questions
  3. Add a #+BEGINPROMPT block with your question
  4. Position cursor in the PROMPT block
  5. Press C-c C-c
  6. Wait for the AI response to appear below

6.2. Configuration

You can configure the AI behavior at the file level using these properties:

  • #+AI_SYSTEM: - System prompt to set AI behavior
  • #+AI_MODEL: - Specify which model to use
  • #+AI_TEMPERATURE: - Control randomness (0.0 to 1.0)
  • #+AI_MAX_TOKENS: - Limit response length
  • #+AI_COMMAND: - Override which AI CLI to use

7. Advanced Usage

7.1. Custom Adapters

You can create custom adapters for any AI CLI tool. Here's an example: 

(defun my-custom-ai-adapter (messages system-prompt model temperature max-tokens)
  "Custom adapter for my-ai-tool."
  (let ((args '("chat" "--json")))
    (when model
      (setq args (append args (list "--model" model))))
    ;; Format messages as JSON and pass to tool
    (append args (list (json-encode messages)))))

;; Use it
(setq org-ai-prompt-args-function 'my-custom-ai-adapter)

7.2. Different AI Tools in Different Files

Each org file can specify its own AI command:

In file1.org: 

#+AI_COMMAND: claude

In file2.org: 

#+AI_COMMAND: gemini

In file3.org: 

#+AI_COMMAND: /usr/local/bin/my-custom-ai-wrapper.sh

Yes, you can customize many of these "variables" through command-line arguments and settings. Here's what's available:

## Command-Line Arguments

When invoking Claude Code from the CLI, you can use:

  • `–add-dir <path>` - Add additional working directories beyond the current directory ```bash claude –add-dir ../apps –add-dir ../lib ```
  • `–model <model>` - Override the model (e.g., `sonnet`, `opus`, or full model name) ```bash claude –model opus ```
  • `–permission-mode <mode>` - Start with a specific permission mode
  • `–allowedTools <tools>` - Specify tools that bypass permission prompts
  • `–disallowedTools <tools>` - Restrict certain tools from use
  • `–output-format <format>` - Set output format (text/json/stream-json)

## Settings File Configuration

In your `settings.json` (at user, project, or enterprise level), you can configure:

### Environment Variables ```json { "env": { "CUSTOMVAR": "value", "CLAUDEBASHMAINTAINPROJECTWORKINGDIR": "true" } } ```

### Additional Working Directories ```json { "additionalDirectories": ["/path/to/other/dir"] } ```

### Custom Status Line (Dynamic Context) ```json { "statusLine": { "command": "/path/to/script.sh" } } ``` This lets you inject custom context into each session.

### Output Customization ```json { "outputStyle": "custom-style-name" } ```

## What You Can't Directly Override

Some values like:

  • Platform/OS (detected automatically)
  • Today's date (system time)
  • Token budget (set by the platform)
  • Git status (read from actual repository state)

These are intentionally read from the actual environment to ensure accuracy.

## For Your Use Case

If you're building a wrapper around Claude Code (which it looks like you might be with `org-dialog-eng`), you could:

  1. Use `–add-dir` to specify additional context directories
  2. Set environment variables via `settings.json` or the `env` configuration
  3. Use a custom `statusLine` command to inject org-mode specific context
  4. Use `–model` to let users choose their preferred model per-file
